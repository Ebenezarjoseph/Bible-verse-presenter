This is "Bible Verse Presenter" — a desktop Electron app for churches.

Purpose: An operator speaks a Bible reference out loud (e.g. "Genesis chapter one verse one" 
or "Genesis one one"). The app listens continuously via microphone, transcribes speech using 
the Azure Cognitive Services Speech SDK (streaming), detects a Bible reference in the transcript, 
fetches the verse text from the free bible-api.com REST API, and displays it in a separate, 
undecorated "Display Window" that the operator drags onto a projector or second monitor.

Architecture:
- Electron main process: owns app lifecycle, creates two BrowserWindows (Control Window, 
  Display Window), handles IPC between them.
- Control Window (renderer): operator UI — start/stop listening button, live transcript log, 
  translation selector, manual verse search fallback, verse history. Runs the Azure Speech SDK 
  directly (it supports microphone input natively in a browser/renderer context via 
  AudioConfig.fromDefaultMicrophoneInput(), no manual PCM streaming needed).
- Display Window (renderer): borderless, fullscreen-able, large-text display of book/chapter/verse 
  and verse text only. No controls visible here.
- Reference parser (pure function, heavily unit-tested): takes a transcript string, returns 
  { book, chapter, verse } or null. Must handle spoken numbers ("one", "twenty-three") as well 
  as digits, and fuzzy-match all 66 book names plus common abbreviations/mishears.
- Verse fetch: calls https://bible-api.com/{book}+{chapter}:{verse}?translation={code}, 
  no API key required.

Conventions:
- Use plain JavaScript (not TypeScript) unless I ask otherwise.
- Keep the reference parser and the API client as standalone modules with no Electron 
  dependencies, so they're independently testable with plain Node/Jest.
- Never hardcode API keys — load from a local .env file (gitignored) via dotenv.
- Prefer small, single-purpose files over large ones.

Version control:
- After every step/feature is working, make a git commit with a clear, conventional message 
  (e.g. "feat: add reference parser with tests") and push to the origin remote.
- Maintain a PROGRESS.md at repo root: one dated entry per step, a one-line description of 
  what was added, and a relative link to a screenshot in docs/screenshots/ showing it working.
- Never commit .env, node_modules, or credentials of any kind.
